# Upgrade — v0.2 (LangChain.js + deepagents)

This release rebuilds the agent runtime on LangChain.js +
`deepagents@1.10`, introduces **Playbooks** + **Learnings** as
first-class primitives, persists conversations, adds an opt-in
**comment-feedback worker**, ships **eval rubrics** and **per-agent
budgets**, and renames the v0.1 **Skill** concept to **Operation**.

The architectural decisions behind it are recorded in
[`docs/internal/adr/0001-langchain-deepagents.md`](../internal/adr/0001-langchain-deepagents.md).
The phased plan is at `/Users/chrisfitkin/.claude/plans/reactive-sleeping-micali.md`.

## TL;DR for upgraders

- `defineSkill` + `Skill` keep working — they're deprecated aliases of
  `defineOperation` + `Operation`. No plugin code needs to change in
  v0.2; rename when convenient.
- Folder layout: `context/<org>/skills/` → `context/<org>/operations/`.
  The loader prefers `operations/` and falls back with a warning.
- New context folders to author:
  - `context/<org>/playbooks/<slug>/SKILL.md` — markdown + YAML
    frontmatter procedural guides agents read on demand.
  - `context/<org>/learnings/<step>.yaml` — whitelisted learning-step
    buckets. Rules added at runtime, never on disk.
  - `context/<org>/evals/<slug>.yaml` — agent eval datasets.
- New env vars:
  - `VOCION_AGENT_RUNTIME=deepagents` — opt into the new runtime. The
    legacy OpenAI loop remains the default until a follow-up flips it.
  - `VOCION_LLM_PROVIDER` / `VOCION_LLM_PROVIDER_MAIN` /
    `VOCION_LLM_MODEL_MAIN` — override per-role model selection.
    Default `main = claude-sonnet-4-6`, `classifier = claude-haiku-4-5-20251001`.
  - `ENABLE_FEEDBACK_WORKER=1` — start the feedback worker.
  - `VOCION_BOT_EMAIL` — anti-loop filter for the worker (skip
    self-authored comments).
- Run migrations: `npm run db:migrate` applies five new migration
  files (0011 – 0016). All additive; no destructive operations.

## What's new

### Agent runtime — deepagents

`runAgentDeep` in `services/AgentService.ts` runs through
`createDeepAgent`. It gives you:

- **Subagents**: declared per-agent in YAML; dispatched via the
  built-in `task("name", "...")` tool.
- **Virtual filesystem**: per-request `StateBackend` seeded with
  playbook bodies and rendered learnings. The agent reads them with
  `read_file` / `glob` / `grep`.
- **TODOs**: built-in `write_todos` middleware. The model plans
  multi-step work before executing.
- **Streaming**: the new SSE route at `/rpc/agent/stream` consumes
  deepagents's `streamEvents(input, { version: 'v3' })` projections
  (`run.messages`, `run.toolCalls`, `run.subagents`) and emits the
  legacy `AgentEvent` shape the UI already understands.

The new route adds 15-second keepalive comments and replays
conversation history from `conversation_message` rows. See
`docs/internal/adr/0001` §2 for the streaming-wire decisions.

### Skill → Operation rename

| Before | After (canonical) |
|---|---|
| `defineSkill(...)` | `defineOperation(...)` |
| `Skill<I,O>` | `Operation<I,O>` |
| `AnySkill` | `AnyOperation` |
| `PluginManifest.skills` | `PluginManifest.operations` |
| `services/SkillService.ts` | `services/OperationService.ts` (re-export) |
| `context/<org>/skills/` | `context/<org>/operations/` |
| MCP `runtime_run_skill` | `runtime_run_operation` (+ alias) |

The deprecated names keep working for one release cycle. v0.3 will
delete them.

### Playbooks

A Playbook is a markdown + YAML frontmatter guide the agent reads on
demand (rather than a typed LLM call like an Operation). Frontmatter
declares slug, name, description, and tags; the body is whatever
procedural guidance you want.

Author at `context/<org>/playbooks/<slug>/SKILL.md`. Sibling files
(`REFERENCE.html`, `COMPONENTS.md`, `examples/*.json`) ride along.
Agents pick up playbooks whose tags intersect their `playbookTags`
field. Empty `playbookTags` = mount everything.

### Learnings

A whitelisted per-step rule store. Author the step in
`context/<org>/learnings/<step>.yaml`; rules are added at runtime by
the agent via `add_learning` (only after the user explicitly
approves a candidate the self-improver subagent proposed). Trigram
Jaccard dedup at 0.72 rejects near-duplicates.

The agent reads the active rules as `/learnings/<step>.md` on each
turn — same surface as rev-ai.

### Conversations

`conversation` + `conversation_message` tables persist threads. The
SSE route's `RunCollector` buffers streamed text + tool breadcrumbs
and writes them on stream completion. Tool entries are dropped when
the history is replayed back to the agent (UI-only ornaments).

Conversations are agent-scoped — the sidebar filters to the active
agent. Auto-titled from the first user message.

### Self-improver + feedback worker

The `self-improver` subagent surveys recent runs + feedback and
proposes candidate rules. It NEVER auto-commits; the main agent
relays candidates to the user for approval, then calls
`add_learning`.

The feedback worker (`npm run worker:serve` / docker-compose `worker`
profile) drains `feedback_job` rows queued by Drive comment
webhooks, classifies them via Haiku (`edit | rule | both | ignore`),
and writes the classification back. Anti-loop bot-author filter via
`VOCION_BOT_EMAIL`.

### Evals

`npm run eval:run -- --dataset sales-assistant-baseline` runs a context-
authored dataset through the agent and scores each case via an LLM
judge. Stamps `contextSha` on every run so prompt changes show as
eval drift. Threshold for CI exit code 0: passRate ≥ 0.8.

oRPC: `evals.list/get/run/runs/runDetail`. UI viewer is intentionally
thin — CLI is the primary surface.

### Budgets

`agent_budget` per `(orgId, agentSlug, period)` with soft + hard
limits in both tokens and USD cents. The Langfuse callback's
`onTurnEnd` hook charges usage from
`usage_metadata.input_tokens/output_tokens` (Anthropic also
surfaces `cache_read_input_tokens` — cache reads bill at 10% per
the model cards).

Pre-flight check in `runAgentDeep` refuses to start a run when over
the hard cap. Budgets are opt-in: no `agent_budget` row → no
enforcement.

Set limits via oRPC `budgets.upsert` (admin-only). UI gauge ships
with the AskChat decomposition in a follow-up.

### HITL gate

`request_human_review` is a universal pause-for-approval tool. The
agent emits a `hitl_gate` SSE event with a name, question, optional
payload (for UI preview) and resumeUrl. The user's next turn carries
their decision; the agent reads its own prior gate request alongside
the new user message to continue or abandon.

Generalizes rev-ai's blueprint-review page. Same primitive backs
workflow-run approve steps.

## Migration checklist

1. `npm install` — picks up the new LangChain + deepagents deps.
2. `npm run db:migrate` — applies migrations 0011 – 0016.
3. `npm run context:apply` — picks up the renamed `operations/`
   directory and any playbooks/learnings/evals you've authored.
4. Set `VOCION_AGENT_RUNTIME=deepagents` to opt into the new
   runtime via the SSE route at `/rpc/agent/stream`.
5. (Optional) Set `ENABLE_FEEDBACK_WORKER=1` and run
   `npm run worker:serve` (or `docker compose --profile worker up`)
   to enable the comment-feedback queue.
6. (Optional) Run `npm run eval:run -- --dataset sales-assistant-baseline` to
   verify the agent runtime end-to-end against the seeded dataset.

## Renaming the reference agent (v0.2 → v0.3)

The demo agent shipped as `ziggy` in v0.1 and v0.2. v0.3 renames it
to `sales-assistant` (display: *Sales Assistant*) to read as a generic
use-case example rather than a proprietary persona. If you have an
existing Vocion deployment with an `agent` row at slug `ziggy`, you
have two options:

- **Rename in place**: `UPDATE agent SET slug='sales-assistant', name='Sales Assistant' WHERE slug='ziggy';`
- **Seed alongside**: just run `npm run context:apply` — a new agent row
  is created at `sales-assistant`. The old `ziggy` row is left intact
  for back-compat; drop it whenever you're ready.

The local-dev Langfuse project id + keys also changed (from
`pk-lf-vocion-ziggy` to `pk-lf-vocion-demo`). If you'd
already provisioned the local Langfuse project, recreate it with the
new id, or update your `.env.local` to point at the existing project's
keys.

## What did NOT change

- Plugin SDK contract — `defineSkill` + `PluginManifest.skills` keep
  working unchanged.
- Onyx integration — `ctx.retrieve` + `search_onyx` are unchanged.
- Stripe billing, Clerk auth, oRPC transport, Drizzle ORM.
- The hand-rolled `runAgent` (OpenAI loop) — still the default until
  the feature flag is flipped in a follow-up.
