# Upgrading from v0.2 to v0.3

The v0.3 release is the biggest framework change since the initial cut. Headline items:

- **Retrieval is now native** — Postgres + `pgvector` + `tsvector`. Onyx is no longer required.
- **Source plugin SDK is GA** — `@vocion/sdk@^0.3` ships the typed Source contract; six first-party connectors ship as the starter pack.
- **`skills/` directory renamed to `operations/`** — the user-facing term changed in v0.2; v0.3 makes it the canonical directory name.
- **LangChain v1 + deepagents agent runtime** is the default.
- **Langfuse observability** is wired into every LLM call with standard trace tagging.
- **Temporal-backed workflow durability** (opt-in; full cutover in v0.4).

Read this guide before running `context:apply` against a v0.2 install.

## Pre-flight

```bash
# From your existing v0.2 install:
npm test         # green
npm run check:types
npm run context:check    # no diffs (you've applied your latest context)
git status -s             # clean
```

Take a Postgres dump:

```bash
pg_dump $DATABASE_URL > vocion-pre-v0.3.sql
```

## Pull v0.3

```bash
git fetch origin
git checkout v0.3.0      # or origin/main if you're tracking HEAD
npm install
```

## Schema migrations

Three new migrations land in v0.3:

| Migration | What it does |
|---|---|
| `0017_perpetual_karnak.sql` | Source SDK tables: `source_definition`, `source_install`, `source_credential`, `source_dek`, `source_audit`. |
| `0018_langfuse_trace.sql` | `conversation_message.langfuseTraceId` column for per-turn deep-links from the chat UI. |
| `0019_pgvector_retrieval.sql` | `CREATE EXTENSION vector` + `knowledge_source`, `knowledge_document`, `knowledge_chunk` with HNSW + GIN indexes. |

```bash
npm run db:migrate
```

The `pgvector` extension requires Postgres 13+ (we recommend 16+) and that the extension is available. Most managed Postgres providers (Neon, Supabase, RDS, GCP) ship it pre-installed. If you're self-hosting, add `postgresql-XX-pgvector` to your base image.

## `skills/` → `operations/` rename

The canonical directory is now `operations/`. The framework still reads from `skills/` with a deprecation warning, but new content should land in `operations/`.

```bash
git mv context/<org>/skills context/<org>/operations
```

Per-operation YAML files keep their existing `skill.yaml` name for v0.3; the file rename is a v0.4 change.

The DB tables (`skill`, `skill_run`) and the public API endpoint (`/api/v1/runs`) keep their names for back-compat. Internally, `SkillService` re-exports as `OperationService`.

## Env vars

New / changed:

| Var | What changed |
|---|---|
| `OPENAI_API_KEY` | Now **required** if any retrieval is configured — the default embedding model is OpenAI `text-embedding-3-small`. Was optional in v0.2. |
| `VOCION_AGENT_RUNTIME` | Set to `deepagents` to opt into the v0.2+ runtime. Default in v0.3 onward. |
| `LANGFUSE_PROJECT_ID` | New. Used for `/dashboard/observability` deep-links. Default: `demo`. |
| `NEXT_PUBLIC_LANGFUSE_BASE_URL` / `NEXT_PUBLIC_LANGFUSE_PROJECT_ID` | New. Used for the "Open in Langfuse" buttons rendered in the browser. |
| `NEXT_PUBLIC_APP_URL` | Now required in production for correct sitemap + canonical URLs. |
| `VOCION_TEMPORAL_ADDRESS`, `VOCION_TEMPORAL_NAMESPACE`, `ENABLE_TEMPORAL_WORKER` | New. Required only when running workflows that opt into Temporal durability. |
| `ONYX_API_URL`, `ONYX_API_KEY` | **Deprecated.** Read [Onyx removal](#onyx-removal) below. |

## Onyx removal

v0.3 ships native retrieval. The 12-container Onyx stack is no longer a required dependency.

If you have data in Onyx today, run the one-shot migration:

```bash
npx tsx packages/core/src/scripts/migrate-onyx-to-pgvector.ts --dry-run
# review the diff, then:
npx tsx packages/core/src/scripts/migrate-onyx-to-pgvector.ts
```

The script:
1. Enumerates every Onyx document set scoped to your org.
2. Re-fetches each document via the matching Source plugin (or via the new `@vocion/source-onyx-bridge` plugin if you don't have an upstream source for that document set).
3. Ingests through `IngestionService` — chunked + embedded + indexed into `knowledge_chunk`.

After the script runs clean:

```bash
docker compose --project-name onyx-stack down -v       # stop + wipe Onyx
rm -rf infra/onyx/onyx-repo                            # remove the cloned upstream
```

Remove the `ONYX_*` env vars from your production env file.

If you had a **custom Onyx connector** (Python code in your Onyx fork), you have two options:

- **Recommended.** Port it as a `Source<Config>` plugin under `packages/plugins/<your-source>/`. The contract is small (`pull(ctx, opts): AsyncIterable<SourceDocument>`); see [Writing a plugin](../guides/writing-a-plugin.md) and `@vocion/source-web-crawl` as the simplest reference.
- **Escape hatch.** Keep your Onyx fork running and use `@vocion/source-onyx-bridge` to pipe its retrieval API through Vocion. Onyx becomes one of N sources, not the framework's retrieval backbone.

## Observability

Run the one-shot bootstrap once per environment so Langfuse can compute USD costs for the new Claude models:

```bash
npm run langfuse:bootstrap
```

This idempotently registers Claude 4.6 / 4.7 / Haiku 4.5 + OpenAI pricing via `/api/public/models`. Subsequent traces show `calculatedTotalCost` in the Langfuse UI. See [Observability](../guides/observability.md).

## Plugin SDK v0.2 → v0.3

The `@vocion/sdk` package adds the `Source<Config>` contract for connectors. Existing Skill plugins are unchanged — bump their peer-dep range to `"^0.1.0 || ^0.2.0 || ^0.3.0"` and you're done.

```jsonc
// packages/plugins/<your-plugin>/package.json
"peerDependencies": {
  "@vocion/sdk": "^0.1.0 || ^0.2.0 || ^0.3.0",
  "zod": "^4.0.0"
}
```

## Verify

After the upgrade:

```bash
npm test                              # 78+ passing
npm run check:types
npm run context:apply                 # idempotent; no errors
npm run docs:lint                     # exits 0
npm run langfuse:smoke                # trace round-trips
```

Open the dashboard, send a chat message to an agent, and verify a trace lands in Langfuse with `tags=[feature:agent.chat, org:<your-org>, slug:<your-agent>]`. If retrieval is configured, run a query that should hit your ingested corpus and watch a child `feature:retrieval.search` trace.

## Rollback

The migrations are forward-only by design. If you must roll back:

1. Restore from the `pg_dump` you took at pre-flight.
2. `git checkout v0.2.X`.
3. `npm install`.

Document any data captured *after* the v0.3 cutover before you blow it away.
